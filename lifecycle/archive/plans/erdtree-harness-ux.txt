## 0. CONTEXT & GOALS

### Problem statement
The 7B tier (Radagon, PRIMARY) was run live and the FELT experience is
rough. The structural harness is solid (1795 tests collected) but:
  - Model text lands as one buffered wall — no streaming, no token-by-
    token feel. The transport ALREADY streams (core/model/ollama.py
    `stream()` yields StreamChunk deltas, ~line 263) but the live path
    collapses them: `chat()` (~lines 296-345) does
    `content="".join(content_parts)` and only THEN does the loop see it.
    The buffering is in OUR code, one layer above a working stream.
  - No feedback while a tool runs — the user stares at nothing between
    "submit" and "wall of text".
  - The small base model flails: clumsy tool picks, verbosity, narration,
    malformed/loose tool calls.

This is a HARNESS-CRAFT problem, not a model problem. Claude Code /
OpenCode make weak-feeling interactions feel good through streaming,
inline step display, disciplined prompts, and instructive self-correction.
Per docs/decisions/0001-framework.md those two are REFERENCE/INSPIRATION
ONLY — never imported, never shipped. We are making OUR harness feel like
theirs, in clean-room code under core/agent/.

### Success criteria (measurable)
  SC1  Model English text renders INCREMENTALLY. Provable on the dev host:
       a scripted CHUNKED responder feeds N content deltas; the IO double
       records >= N incremental render calls (not one joined string).
  SC2  When a tool is dispatched, a live "running: <cmd>" line appears
       BEFORE the result and a result/status line appears after — both
       I2-clean (pass _FORBIDDEN_AI_TERMS).
  SC3  Every NEW user-facing string (progress lines, tool-run display,
       any spinner/status text, new re-ask wording) passes the I2 filter
       (core/agent/prompt.py `_AI_PATTERN`). A test asserts this over the
       full new-string inventory.
  SC4  The permission gate, audit records, and dead-man fallback are
       BEHAVIORALLY UNCHANGED: a streamed token NEVER represents an
       unconfirmed write; the gate fires at the same point on the same
       inputs; audit record count/content is identical to pre-change for
       the same scripted turns. Pinned by a regression/golden test.
  SC5  The existing buffered responder + ConsoleIO path still works
       unchanged (backward compatible). Full suite stays green
       (>= 1795 tests; new tests added).
  SC6  Small-model discipline: the house prompt carries explicit
       when-to-call-a-tool guidance + a short few-shot tuned for a 7B/3B,
       and the re-ask is tighter/more instructive. Validity is measured by
       bench/ (separate track) — this plan delivers the PROMPT + the
       TESTABLE re-ask wording, not a fine-tune.

### Non-goals (explicitly out of scope)
  - No model fine-tuning. Live-model "feel" on the 7B/14B is partly
    DEFERRED to the benchmark/fine-tune track (bench/, Phase-10 per 0001).
    This plan makes the HARNESS good enough that a weak model feels good;
    it does not raise raw model validity numbers.
  - No live Ollama round-trip claim. The dev host has no Ollama/Linux
    (DEV-HOST NOTE in ollama.py). Live streaming feel is reasoned + unit-
    proven with doubles; any real round-trip is DEFERRED-TO-MOSSAD.
  - No change to the 0002 wire contract, the tool-call protocol, the
    permission classifier (permissions.py is BYTE-UNTOUCHED), or the audit
    schema.
  - No new dependency. stdlib only (the project's stated constraint;
    ollama.py uses urllib by design).
  - No async/event-loop rewrite. The loop stays synchronous; streaming is
    a synchronous generator drained in the loop.

### Assumptions made
  A1  The model seam stays the injected `responder` callable. We ADD a
      second, OPTIONAL streaming seam rather than replacing it, so every
      existing test that injects a ScriptedResponder keeps passing
      (test_repl.py). The buffered `responder` remains the default.
  A2  ReplIO is a Protocol (repl.py:65). We extend it with an OPTIONAL
      incremental hook; ConsoleIO and FakeIO (test_repl.py:54) implement
      it; absence of the hook degrades to today's `render(text)` behavior.
  A3  Streaming is presentation-only. The loop still assembles a full
      AssembledResponse-equivalent BEFORE routing/gating/dispatch. Tokens
      are surfaced to the IO as they arrive, but the router/gate/audit see
      the SAME assembled turn they see today. This is the load-bearing
      safety property behind SC4.
  A4  "Live tool-call display" shows the SYNTHESIZED command string
      (synthesize_command, repl.py:137) — which is already I2-clean shell
      argv (systemctl/dnf/...) — not the tool name or any model framing.
  A5  Few-shot examples go in the house system prompt (prompt.py); they
      are checked by `_assert_no_ai_language` at import time like the
      existing _HOUSE_SYSTEM_PROMPT (prompt.py:113), so any I2 leak fails
      the build immediately.
  A6  Tests run via /home/aaron/erdtree/.venv/bin/python -m pytest (NOT
      system python3).

## 1. ARCHITECTURE OVERVIEW

### How this fits
Three independent quality dimensions over the SAME loop spine
(repl.py `run_turn`):

  (1) STREAMING SPINE  — thread `stream()`'s deltas to the IO without
      collapsing them in our code. Touches ollama.py (add a streaming-
      capable responder), repl.py (drain a stream while assembling),
      ReplIO (incremental hook), shell/main wiring.
  (2) LIVE TOOL STEPS  — surface the gate decision + synthesized command
      + result as inline status lines via the IO. Sits in _dispatch_calls.
  (3) PROMPT DISCIPLINE — prose-only work in prompt.py (when-to-call
      guidance, few-shot) + tighter re-ask wording in router.py.

Dimension (1) is the SPINE; (2) depends on the IO surface (1) introduces.
Dimension (3) is INDEPENDENT (pure prose/string work) and can land in
parallel with (1)/(2).

### Affected services
  - core/model/ollama.py   — add a streaming responder adapter (additive)
  - core/agent/repl.py     — ReplIO incremental hook + stream drain +
                              tool-step display (SINGLE-WRITER, serialize)
  - core/agent/prompt.py   — tool-use guidance + few-shot (prose)
  - core/agent/router.py   — tighter, more instructive re-ask wording
  - core/agent/main.py     — wire the streaming responder + streaming IO
  - shell/shell.py         — unchanged in behavior; verify the dead-man
                              guard still wraps the (now-streaming) turn

### Data flow (after this plan)
```
user line
  -> shell.run_turn / Repl.run_turn
  -> assemble(prompt + few-shot + context + tools)        [prompt.py]
  -> STREAMING responder: for chunk in client.stream(...):
        if chunk.content_delta -> io.render_delta(token)  [NEW hook]   *** presentation only
        accumulate content + tool_call_deltas
     -> returns ONE AssembledResponse (same shape as today)            *** router sees assembled turn
  -> router.route(content, tool_calls)                    [router.py]
  -> per call: synthesize_command -> permissions.classify  [UNCHANGED gate]
        io.tool_step("running: <cmd>")                    [NEW, I2-clean]
        gate resolve -> dispatch -> audit.write           [UNCHANGED]
        io.tool_step_result(status)                       [NEW, I2-clean]
  -> tool results fed back; loop; English answer streams to terminal
```
The gate/audit/dead-man boxes are byte-untouched. The only NEW edges are
the dotted "presentation only" render_delta + tool_step lines.

### Invariants this plan must preserve
  I1  Streaming still talks ONLY to localhost Ollama. ollama.py already
      asserts loopback at construction (_assert_localhost). The streaming
      responder reuses the SAME OllamaClient — no new socket, no new host.
  I2  Every NEW user-facing string passes _FORBIDDEN_AI_TERMS (no
      ai/llm/model/agent/agentic/inference/ollama/...). Enforced by test
      and, for prompt strings, by _assert_no_ai_language at import.
  I3  Permission gate resolved BEFORE every write/destructive dispatch,
      via the UNCHANGED permissions.classify. A streamed token is content
      ONLY; tool calls are assembled and gated exactly as today. NEVER
      stream a write's effect before the gate clears.
  I4  EVERY attempted op writes exactly one audit record — unchanged.
      tool-step display is NOT an audit substitute and does not alter
      audit calls.
  I9  Dead-man fallback (shell/shell.py) byte-untouched; streaming must
      not introduce a new unbounded wait or swallow ConnectionError.
