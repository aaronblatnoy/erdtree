## 0. CONTEXT & GOALS

Problem statement
  Every component of the local RAG pipeline EXISTS and is tested in isolation
  (51 rag tests pass in the project venv), but nothing connects them on REAL
  sources end to end. The current state of the pipeline:

    build_corpus()  --writes-->  corpus.jsonl     [reader for man + rocky docs OK]
         |  ... GAP: no production glue runs the next two steps on real sources
    embed_texts()   --produces-> vectors          [hashed embedder, offline OK]
    build_index()   --writes-->  corpus.db        [sqlite-vec single file OK]
    retrieve()      <--reads--   corpus.db        [frozen engine OK]
    docs.retrieve   --calls-->   retrieve()       [env-knob index path; degrades OK]

  The ONLY production glue that exists is rag/fixtures/build_fixture_index.py,
  which is HARDCODED to the 12-chunk hand-curated test fixture. The CLI in
  build_corpus.__main__ writes a JSONL but stops there (no embed, no index).
  So today there is NOTHING between "writes a fixture jsonl" and "builds a
  fixture .db" that runs on real man pages + real Rocky docs and produces a
  shippable index the docs tool can point at. That single orchestrator is the
  deliverable.

## 1. ARCHITECTURE OVERVIEW

How this fits into the existing system
  This plan adds exactly ONE new production module (the orchestrator) plus a
  corpus-source staging step, a config default, a decision doc, and tests. It
  sits in the rag/ layer, OUTSIDE core/, and never adds a runtime import to the
  core agent. The core agent's only contact with the index remains the existing
  read-only env knob ERDTREE_CORPUS_INDEX consumed by core/tools/docs.py.

  Data flow (build-time, offline, no network — I1):

    durable corpus sources                  rag/build_index_prod.py
    ----------------------                  (THE NEW ORCHESTRATOR)
    /usr/share/man  (GPL-2.0) -----+
                                   +--> build_corpus() --> corpus.jsonl
    <durable>/rocky-docs/  --------+         |
       (CC-BY-SA-4.0, filtered)              v
                                        embed_texts(backend=hashed|st)
                                             |
                                             v
                                        build_index() --> corpus.db  (sqlite-vec)
                                             |                 single file
                                             v
                                        write LICENSES manifest + meta record

  Data flow (runtime, the only core contact — already exists, unchanged):

    ERDTREE_CORPUS_INDEX=<corpus.db>  --> core/tools/docs.py _op_retrieve
                                            --> rag.retrieve.retrieve(...)
                                            --> "[source]\n<passage>" to the loop
    (absent/unreadable index -> "Retrieved 0 reference passages." — I9 degrade)

Affected layers
  rag/        : NEW orchestrator + a source-staging helper + LICENSES regen.
  docs/decisions/ : NEW 0004 (license posture). 0003 stays frozen, untouched.
  core/       : NO code change. A documented config DEFAULT for the index path
                is the only contact; ERDTREE_CORPUS_INDEX plumbing already exists.
  tests/      : NEW real-content smoke test + retrieval-quality test +
                episodic-reuse regression. All run under .venv unittest.
  scripts/ or docs : a reproducible "stage sources" recipe + a "build index"
                invocation, documented for both dev host and firstboot.

Invariants this plan must preserve
  I1  No network at build OR runtime. build_corpus/embed/index already open no
      socket; the orchestrator adds none. The Rocky docs are already on disk
      (git clone done out-of-band, once); the orchestrator never fetches.
  I2  No AI/LLM/model/inference/embedding/retrieval terms in any USER-FACING
      string. Chunk TEXT feeds the model prompt (acceptable — it is reference
      content), but any operator-facing artifact (the source label, a build
      summary printed to a console an operator might see, the LICENSES manifest
      headers) stays I2-clean. The docs tool summary ("Retrieved N reference
      passages.") is untouched. A test greps the orchestrator's stdout + the
      source labels of produced chunks against core.agent.prompt._FORBIDDEN_AI_TERMS.
  I6  The index path stays an OPAQUE env knob (ERDTREE_CORPUS_INDEX); the
      orchestrator does not bake a tier name into the path or the artifact.
  I9  docs.retrieve still degrades empty-but-valid when the index is absent —
      the production index makes it NON-empty; the degrade path is untouched.
  0003 FROZEN: sqlite-vec backend, single-.db format, retrieve() signature,
      Chunk shape, lexical rerank. Additive only. Episodic reuse via a SECOND
      index_path must keep working.

### PHASE 2: The production orchestrator — GOAL: corpus -> embed -> index in one command
--------------------------------------------------------------------
   Prerequisites: Phase 1 (durable sources exist).
   Deliverables:
     - rag/build_index_prod.py with:
         run_build(*, man_dirs, rocky_doc_dirs, output_index, output_jsonl,

       embed_texts(texts, backend=, dim=, model_name=), calls build_index(
       output_index, chunks, vectors, dim=, embed_backend=, embed_model=),
       regenerates rag/LICENSES.md, and returns a report (chunk count, per-source
       counts, index bytes, elapsed).
     - A __main__ CLI mirroring build_corpus's CLI style: --man-dir,
       --rocky-dir, --output-index, --output-jsonl, --backend, --dim,
       --max-man-pages, --max-rocky-files.
     - OPTIONAL additive meta keys (built_at, source_counts) if Phase 2 decides
       they are worth it (additive, retrieve ignores unknown keys).
   Steps:
     1. Implement run_build composing the three frozen functions. Do NOT
        reimplement chunking/embedding/indexing — call the existing APIs exactly.
     2. Map CorpusChunk: build_corpus returns RawChunk (chunk_id,source,license,
        text); rag.index.CorpusChunk is the same 4 fields — convert 1:1 (the
        fixture builder is the template; generalize it from hardcoded JSONL to a

        printed strings stay clean.

   Risks / Edge cases:
     - sqlite_vec absent (wrong interpreter) -> orchestrator must run under .venv;

     - Embedder swap (hashed->st) must not change the index FORMAT — it doesn't;
       embed_backend/dim are recorded in meta and retrieve reads them back.
   Subagent orchestration:
     - Execution shape: [SEQUENTIAL — depends on Phase 1]
     - REQUIRED subagent_type: general-purpose
     - REQUIRED model: sonnet  (compose frozen APIs; the fixture builder is the
       template — moderate, not frontier, reasoning)
     - Run in background: no (Phase 3's test gates on this artifact)
     - Brief template:
       "Execute Phase 2 of <PLAN_FILE_PATH>. Your slice: write rag/build_index_prod.py

## 4. DATA MODEL CHANGES
   None to the frozen format. The sqlite-vec .db (meta + chunks + vec_chunks)
   is unchanged. OPTIONAL additive meta keys (built_at, source_counts, corpus_sha)
   may be written by the orchestrator — additive, ignored by retrieve.read_meta,
   recorded in 0004 (NOT 0003). No SQL migration; build_index recreates the file.
   No RLS / encryption concerns (single-machine local file; no PII; chunk text is
   public documentation).

## 5. API CONTRACT
   No HTTP API. The "contract" surfaces are:
     - rag.build_index_prod.run_build(...) -> BuildReport  (NEW; internal build API)
     - rag.retrieve.retrieve(query, index_path, k, max_chars) -> list[Chunk]
       (FROZEN — unchanged; the orchestrator's output is consumed through it)
     - core/tools/docs.py docs.retrieve op (FROZEN ToolSpec — unchanged; only the
       index it points at via ERDTREE_CORPUS_INDEX becomes non-empty)

## 6. WORKER / QUEUE CHANGES
   None. The corpus build is an OFFLINE, one-shot build-time job (dev host now;
   firstboot on target under BRANCH B). No queue, no scheduler, no daemon. The
   full st-embedder corpus build is a separate DEFERRED-TO-MOSSAD batch job (D1),
   off the critical path.

## 7. FRONTEND CHANGES
   None. Erdtree is terminal-native, no GUI. The only operator-visible surface is
   the docs.retrieve tool summary ("Retrieved N reference passages.") which is
   unchanged, and the orchestrator's I2-clean build summary printed at build time.

## 8. TESTING STRATEGY
   Unit / integration (tests/test_rag_build_index_prod.py, .venv unittest):
     - Smoke: run_build over capped man + staged Rocky docs -> real .db opens,
       meta sane, chunk count > 0, per-source split has both man + rocky chunks.
     - Retrieval-quality (SC2): "how do I open a firewall port" -> top result
       source contains "firewall", text mentions port/zone. The load-bearing
       assertion; proven feasible this session.
     - Episodic-reuse regression (SC6): a second index at a different index_path
       (synthetic chunks, license="") builds + queries independently.
     - I2 leakage: orchestrator stdout + chunk source labels vs _FORBIDDEN_AI_TERMS.
     - I9 degrade: empty ERDTREE_CORPUS_INDEX -> "Retrieved 0 reference passages."
     - No-regression: the 51 existing rag/docs tests stay green.
   Manual QA checklist:
     - Stage sources, run the orchestrator CLI, export ERDTREE_CORPUS_INDEX, call
       docs.retrieve in a Python shell, confirm a grounded firewalld passage.
     - DEFERRED-TO-MOSSAD: full uncapped man build + st-embedder build + live
       latency on an 8 GB card (D1/D2 below).

## 9. ROLLOUT & OBSERVABILITY
   - No feature flag needed: ERDTREE_CORPUS_INDEX unset -> docs tool degrades OFF
     (I9). Setting it to the durable corpus.db turns retrieval ON. This IS the flag.
   - Observability: the orchestrator prints an I2-clean build summary (passage
     count + size). The optional meta keys (built_at, source_counts) give an
     operator provenance via a plain `sqlite3 corpus.db "select * from meta"`.
   - Rollback: delete/unset ERDTREE_CORPUS_INDEX -> instant return to the
     degraded empty-but-valid state; the core agent loses retrieval but keeps
     running (I9). The index is a single file; removing it is the rollback.

       P1 ──> P2 ──> P3 ──┬──> P4  (owner gate; does NOT block P5)
                          └──> P5  (measure + runbook)

   is absent, P4 pauses while P5 still completes.

### 12.2 Cross-phase work consolidation
   - The corpus.db built in P2 is REUSED by P3 (tests) and P5 (measurement).
     Build it ONCE at runtime/rag/corpus.db; P3 + P5 read that artifact rather
     than each rebuilding. (P3's smoke test builds its OWN tmp index for
     isolation, but the firewall retrieval-quality + P5 measurement use the
     shared runtime/rag/corpus.db.)
   - .venv/bin/python is the interpreter for EVERY phase that touches an index
     (P2/P3/P5). The executor should export the venv once and pass it in briefs;
     the #1 footgun is running system python3 (no sqlite_vec -> silent no-index).
   - rag/LICENSES.md is regenerated by BOTH the orchestrator (P2) and reconciled
     in P4 — run P4's LICENSES touch AFTER P2/P5 to avoid clobbering; P4 and the
     orchestrator both call write_license_manifest, so P4 only edits the
     posture-specific note, not the whole file.

### 12.3 Model-mix budget
   Target split across the 5 in-scope phases:
     - fable:  0 phases. Nothing here is frontier-grade — it is composing already-
       built, tested, frozen functions and writing tests + docs. Opus on P4 is
       the ceiling.
     - opus:   1 phase (P4 — two-branch license reasoning + a recommendation the
       owner acts on). ~20%.
     - sonnet: 4 phases (P1/P2/P3/P5 — file logic, API composition, tests,
       measurement). ~80%.
     - haiku:  0 phases (no phase is a trivial single-field edit; P1 is the
       lightest but still has filter logic + idempotency to get right).
   Rough split: 0% fable / 20% opus / 80% sonnet / 0% haiku.

### 12.4 Estimated wall-clock
   Best case (P4 ‖ P5 after P3, owner immediately available): P1+P2+P3 serial
   spine (~the bulk) + max(P4,P5) ≈ one focused session.
   Worst case (strictly serial, owner-gate wait on P4): the spine + sequential
   P4 (incl. owner round-trip) + P5.
   Critical path: P1 -> P2 -> P3 (build + prove). P4's owner gate is the only
   unbounded wait, and it is OFF the index-correctness path (the index works
   before the posture is decided).

### 12.5 Failure recovery policy
   - P1 (staging): retry up to 2x with same subagent (deterministic file ops);
     if source clone is missing, PAUSE + escalate (need the Rocky docs on disk).
   - P2 (orchestrator): retry up to 2x. If build_index raises "sqlite-vec backend
     not installed", the executor mis-set the interpreter -> re-dispatch under
     .venv/bin/python (do NOT treat as a code bug).
   - P3 (tests): if the firewall retrieval-quality assertion FAILS (not skips),
     PAUSE + escalate — it signals the hashed embedder under-ranked the Rocky doc
     for the chosen corpus scope; the fix is corpus scope (Q2/Q3) or the
     st-embedder swap (D1), an owner-visible quality decision, not a silent retry.
   - P4 (license gate): PAUSE + escalate to owner by design — never self-accept a
     posture.
   - P5 (budgets/runbook): retry up to 2x; non-blocking — if it fails, log a
     deferred-item and ship the index (the runbook is documentation, not the gate).
   - Default if unspecified: pause and escalate.
