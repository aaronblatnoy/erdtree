## 0. CONTEXT & GOALS

Problem statement
  Sysadmins and homelabbers interact with Linux through a command line that
  demands memorized syntax, man-page archaeology, and zero forgiveness. Erdtree
  replaces that surface with English while keeping the system fully open
  underneath: "Type in English. Linux does it." The intelligence is local-only
  (Ollama), invisible (never surfaced as AI/LLM/agent/model), and gated by a
  deliberate permission seam because the domain is irreversible (a live box has
  no git-undo). This is "Claude Code on hard mode": same UX target, a weaker
  local engine, an unforgiving domain, an eternal login-shell session, and a
  hidden-AI reliability bar where users blame "the shell," not "the AI."

Success criteria (measurable)
  SC1  End-to-end agent loop works on a base model (Qwen2.5 7B/14B-Instruct)
       under Ollama: English in -> tool call -> permission gate -> subprocess
       execute -> audited -> English out. No external network calls at runtime
       (verified by egress monitor).
  SC2  Tool-call validity rate >= 99.5% on the reliability benchmark for the
       Radagon base model; documented (pass/fail) for the Marika 3B base.
  SC3  The product shell is the login shell AND a dead-man's bash fallback
       fires automatically + loudly when the agent/Ollama is unavailable
       (verified by killing Ollama and confirming the user still has a shell).
  SC4  Already-valid Linux commands (df -h, cd, systemctl status, grep ...)
       run transparently and are NEVER mistranslated; English intent is
       translated. Measured by a command-vs-English dispatch test corpus
       (target: 0 mis-dispatches on the corpus).
  SC5  Read ops execute with no confirm; write ops confirm; destructive ops
       require the literal word typed in full and are never auto-confirmed,
       never run non-interactively. Every op lands in append-only JSONL audit.
  SC6  Two bootable, branded ISOs (Marika, Radagon) install on target hardware
       and first-boot into the product shell with the model present (or
       gracefully pulling, with bash fallback live during the pull).
  SC7  ERDTREE_TIER selects tier at runtime; core/ contains zero hardcoded
       "marika"/"radagon" strings (verified by grep). Rocky branding never
       appears in any user-facing string (verified by grep).

Non-goals (explicitly OUT of scope for v0.1 — DEFERRED)
  - Radahn (~70B) and Starscourge tiers. Do not build.
  - Fleet / multi-node operational RAG corpus (a distributed-systems problem;
    Radahn-era). v0.1 RAG is single-machine + shared docs only.
  - Any GUI. Terminal only, always.
  - Shipping the FINE-TUNED model in the initial agent. v0.1 ships on a base
    model + strong system prompt. The fine-tuned Radagon model is a PARALLEL
    workstream that lands in v1.0 and does NOT block v0.1.
  - Any external/cloud runtime dependency of any kind.

Assumptions made
  A1  Target dev/build host is the "mossad" server (Threadripper-class CPU +
      multiple RTX 3060 Ti 8GB GPUs per CLAUDE.md / design notes) for training
      and ISO builds. Per-card 8GB is a hard constraint: a 14B 4-bit (~9GB)
      does NOT fit on one card — Radagon inference must split across cards or
      drop quant. Captured in Phase 6/8 risks.
  A2  The "Claude Code-style harness" is an available open-source agent harness
      that speaks an OpenAI-compatible endpoint; Ollama exposes one. Its exact
      identity/license is unresolved and is an EARLY GATING item (Phase 0).
  A3  Base model for v0.1 is Qwen2.5-Instruct (7B/14B for Radagon, 3B for
      Marika), pinned by quantization tag in tier config (never :latest).
  A4  Python is the agent-layer language; tools shell out via subprocess.
  A5  Rocky Linux 9 is the build base; SELinux enabled; dnf/RPM packaging.
  A6  No git repo exists at the erdtree root yet (verified). Executor should
      `git init` before fan-out so worktree-isolation phases work.
  A7  The harness can be pointed at Ollama without code changes beyond config
      (base URL + model name). If it cannot, Phase 3 absorbs an adapter shim.

If any of A1-A7 is wrong in a way that reshapes the plan, that surfaces in the
Phase-0 gating spike and the plan is revised before further fan-out.

## 1. ARCHITECTURE OVERVIEW

How this fits / data flow (the agent loop, target behavior):

  product shell (login shell)                         os/ integration
        |  user types English OR a raw command        (journald interceptor,
        v                                              dmesg watcher, pam hook,
  shell.py dispatch ----+---- already-valid cmd? ---> passthrough.py (bash)
        |               |                                    |
   English intent       +---- `!` raw escape --------------> bash
        v
  context/collector.py  -> snapshot.py (live system state) -> cache.py
        v
  core/agent/prompt.py  (system prompt + tier prompt + fresh context + history)
        v
  core/model/ollama.py  (stream from LOCAL Ollama, OpenAI-compatible)
        v
  core/agent/router.py  (parse tool calls from stream)
        v
  core/agent/permissions.py  (read=instant | write=confirm | destructive=typed)
        v
  core/tools/*  (subprocess: systemctl, dnf, journalctl, ...)
        v
  core/agent/audit.py  (append-only JSONL -> /var/log/{tier}/audit.jsonl)
        v
  model interprets tool output, responds in English -> loop
        ^
        +--- core/context/  re-derives state on demand (live box = source of truth)
        +--- RAG-as-a-TOOL (agent calls retrieval only when it decides it needs it)

Affected services / layers (all NEW; repo is a fresh scaffold):
  - core/agent/    agent harness glue: repl, router, context, permissions, audit, prompt
  - core/tools/    sysadmin toolset (subprocess wrappers)
  - core/model/    Ollama streaming client
  - core/context/  live system snapshot collector + cache
  - shell/         the login shell + bash passthrough + dead-man fallback + hooks
  - os/            systemd units, journald interceptor, dmesg watcher, pam hook
  - rag/           shared docs corpus build + retrieval tool (single-machine scope)
  - tiers/marika/, tiers/radagon/  config.yaml + prompt.txt + welcome + tools.yaml + firstboot
  - training/      synthetic gen, clean, format, finetune (Unsloth), eval  [PARALLEL]
  - build/, installer/, distro/rpm/  ISO + RPM packaging  [LAST]
  - bench/         tool-call reliability benchmark  [EARLY gating]
  - tests/, .github/workflows/, docs/

Invariants this plan must preserve (thread through EVERY phase — see §0 SCs):
  I1  No external API calls at runtime. Local Ollama only. (egress test gates)
  I2  AI is invisible: no "AI/LLM/model/agent/agentic" in user-facing strings.
  I3  Permission check before every write/destructive op; destructive never
      auto-confirmed, never non-interactive.
  I4  Audit-log every operation as append-only JSONL.
  I5  System context always injected — agent never asks the user to explain env.
  I6  One codebase; tier differences live ONLY in tiers/ (config + prompts).
      core/ never hardcodes a tier/product name. ERDTREE_TIER selects at runtime.
  I7  Rocky branding never user-visible.
  I8  Performance is a feature — simple ops feel instant (read-path latency budget).
  I9  DEAD-MAN'S FALLBACK: if the agent can't start (crash / Ollama down / model
      mid-pull), fall through to bash automatically and loudly. A headless box
      must never be left shell-less.

### PHASE 1: Safety & System-Awareness Core
            Goal: the agent can SEE the system and CANNOT act unsafely.
=====================================================================
Prerequisites: Phase 0 GO.
Deliverables:
  - core/context/collector.py — gathers live state: distro/kernel, installed
    packages, running services, hardware topology, recent logs, open ports,
    firewall rules, disk health, recent agent changes.
  - core/context/snapshot.py — typed snapshot object the prompt layer consumes.
  - core/context/cache.py — short-TTL cache so repeated reads feel instant (I8),
    but the live box stays source of truth (design decision #5: re-derive on
    demand over stale cache; cache is a latency optimization only).
  - core/agent/permissions.py — classifies an op as read / write / destructive;
    enforces: read=instant, write=confirm, destructive=literal-word-typed-in-full,
    never auto-confirm, never non-interactive. Returns a decision + reason.
  - core/agent/audit.py — append-only JSONL writer: timestamp, NL input,
    translated command, output, result, permission decision. fsync-on-write.
Steps:
  1. Implement collector reading from /proc, /sys, systemctl, rpm -qa, ss, etc.
     No tier names anywhere (I6). Tolerate missing subsystems gracefully.
  2. Define snapshot schema; make it cheap to serialize into the prompt.
  3. Implement permission classifier with an explicit destructive taxonomy
     (rm -rf, mkfs, dd, partition ops, user/SSH/firewall lockout risks, reboot
     on remote, etc.). Default-deny on ambiguity; classify unknown writes as
     write-confirm, unknown destructive-shaped as destructive.
  4. Implement audit writer; append-only, atomic, survives crash mid-write.
Validation:
  - test_snapshot: collector returns a populated snapshot on this box.
  - test_permissions: read ops pass instantly; a curated destructive corpus is
    ALWAYS gated and NEVER auto-confirmable; non-interactive destructive is
    refused. [gates I3]
  - test_audit: every classified op produces exactly one JSONL line; file is
    append-only and parseable; partial-write recovery verified. [gates I4]
Risks / Edge cases:
  - Misclassifying a destructive op as a write (under-gating) = catastrophic on
    a live box. Mitigation: err toward MORE gating; the destructive corpus is a
    living test fixture; route the classifier to adversarial review (see below).
  - Collector latency blows the instant-feel budget. Mitigation: lazy per-domain
    collection + cache; only collect what a query needs.
Subagent orchestration:
  - Execution shape: [PARALLEL FANOUT OK — 3 siblings] context/, permissions.py,
    audit.py are disjoint files. permissions.py is the keystone.
  - REQUIRED subagent_type: general-purpose
  - REQUIRED model: sonnet for context/ + audit.py; opus for permissions.py
    (the safety classifier is correctness-critical; under-gating is irreversible).
  - Run in background: yes (sibling builders), then join.
  - VERIFICATION: route permissions.py + its destructive corpus to the
    audit-duo SKILL — adversarial "find a destructive op that slips the gate."
  - Brief template (per sibling):
      "Execute Phase 1 of <PLAN_FILE>. Your slice: <core/context/* | permissions.py
       | audit.py>. Read §3 Phase 1 + §1 invariants I3/I4/I5/I8. Touch only your
       file(s) + its test. No tier names (I6). Validation: <named test> green.
       Evidence: test output. Print one-line summary and exit."

### PHASE 5: The Product Shell + OS Integration + Dead-Man's Fallback
            Goal: the loop becomes the LOGIN SHELL, safely, with bash fallback.
=====================================================================
Prerequisites: Phase 4 (a working loop to wrap).
Deliverables:
  - shell/shell.py — the product shell (the login shell). CORE dispatch:
    already-valid command -> execute transparently; English intent -> translate;
    NEVER mistranslate (SC4, decision #6). No "AI mode" to enter.
  - shell/passthrough.py — the `!` raw-bash escape (opt-in; default is product
    shell).

  - os/journald/ + os/dmesg/ — interceptor + watcher (feed context/proactive).

    shell-less.

  1. Implement the command-vs-English dispatch. Heuristic + validation: does the
     input parse as a valid command / resolve on PATH / match known syntax? If
     yes, run it raw. If clearly English intent, translate. Ambiguous -> prefer
     safe behavior (don't silently run a guessed command). Build a test corpus.

     banner explaining the degraded state. Must survive Ollama-down and
     model-mid-pull.
  4. os/ integration: systemd ordering so the shell waits for Ollama but does
     NOT hang the login if Ollama never comes up (fallback fires instead).
Validation:
  - test_dispatch: the command-vs-English corpus has 0 mis-dispatches (SC4) —
    raw commands run raw, English translates, ambiguous never silently guesses.
  - test_deadman: kill Ollama / simulate agent crash / simulate model mid-pull
    -> user lands in bash with a loud banner, never shell-less (SC3, I9).
Risks / Edge cases:
  - A mistranslated raw command on a live box = the cardinal failure ("the shell
    is broken," decision #8). Mitigation: dispatch errs toward executing raw or
    asking, never toward guessing; this corpus is a permanent regression gate.
  - Fallback that's quiet = a headless admin can't tell why English stopped
    working. Mitigation: fallback MUST be loud and explain the degraded state.
  - Login shell that hangs on Ollama startup = boot hang. Mitigation: timeout +
    fallback, never an unbounded wait.
Subagent orchestration:

    the keystones. shell/ and os/ are somewhat separable but the dead-man path

  - REQUIRED subagent_type: general-purpose
  - REQUIRED model: opus (the command-vs-English dispatch + dead-man fallback are
    correctness-critical and the felt-UX core of "Claude Code on hard mode"; a

  - Run in background: no (gates the rest of the integrated build).
  - VERIFICATION: route the dispatch corpus + the dead-man fallback to the

### PHASE 8: Context Cycling / Compaction / Invisible Memory UX
            Goal: an immortal session that NEVER shows a context limit.
=====================================================================
Prerequisites: Phase 4 (loop), Phase 1 (audit log = durable transcript),
Phase 7 (RAG-over-own-history is the same retrieval machinery). Decision #5.
Deliverables:
  - core/agent/memory.py — task-scoped (not session-scoped) context; rolling
    compaction that KEEPS tool-call OUTCOMES and DROPS verbose raw outputs once
    reasoned over; keeps recent turns verbatim (deixis: "restart it" / "the one
    we just did"); compaction threshold is a per-tier config knob.
  - core/context/facts.py — a tiny curated per-host facts file injected as a
    preamble (episodic anchor).

  1. Implement task-boundary context cycling: cycle the window INVISIBLY at task
     boundaries; re-inject minimal fresh state per task; don't retain stale state.

  3. Implement per-host facts preamble + audit-log retrieval as episodic memory.
  4. Design against the "amnesia moment": never say "out of context," never ask
    the user to re-explain something established, never show a reset. Reaching

  - REQUIRED subagent_type: general-purpose
  - REQUIRED model: opus (the invisible-memory UX is subtle architecture where a

## 4. DATA MODEL CHANGES
No database. "State" lives in three places (decision #5): (a) the live box
(source of truth, re-derived on demand), (b) the append-only audit JSONL at
/var/log/{tier}/audit.jsonl (durable transcript + episodic memory via RAG), (c)
the tiny per-host facts file (preamble). The model's context window is
disposable per-task scratch. Audit JSONL schema (Phase 1): {ts, tier, nl_input,
translated_command, tool, args, permission_decision, exit_code, stdout_summary,
stderr_summary, result}. Append-only, fsync-on-write, parseable.

---------------------------------------------------------------------
## 5. API CONTRACT
No external HTTP API (I1). The only "endpoint" is the LOCAL Ollama
OpenAI-compatible endpoint (localhost), consumed by core/model/ollama.py.
Internal contracts that matter:
  - Tool-call protocol (docs/decisions/0002): the frozen I/O format the model
    emits, router.py parses, and training data matches. THE central contract.
  - Tool interface (core/tools/__init__.py, Phase 2): name, args schema,
    per-op permission class, execute()->structured result.
  - Tier config (config.yaml): model tag, prompt path, tools allowlist,
    retrieval budget, compaction threshold.

long-running background processes are os/ watchers (journald interceptor, dmesg
watcher) feeding context, and (at install) the firstboot model pull.

### 12.5 Failure recovery policy
  - P0 (gating): on license NO-GO or incomplete telemetry strip -> STOP and
    escalate to the user. This reshapes the plan; do not improvise past it.
  - P1 permissions / P5 dead-man + dispatch / P11 ship-gate invariants: on
    validation failure -> retry once, then PAUSE and ESCALATE. These are
    safety/invariant-critical; never auto-skip a failing safety gate.
  - P2, P6, P7 tool/RAG siblings: on failure -> retry up to 2x with the same
    subagent (non-flaky expected), then log a deferred-item and continue (a
    single missing non-core tool does not block the loop).
  - P4 benchmark below 99.5%: do NOT block v0.1 (base-model ship is acceptable
    by decision) — record the gap loudly as a deferred-to-P10 item.
  - P10 (training): non-blocking by design; on eval-gate miss, v0.1 keeps the
    base model and P10 iterates. Never blocks the v0.1 ship.
  - Any audit-duo split that does not resolve -> PAUSE and ESCALATE with the
    honest unresolved split (no manufactured consensus).
  - Default if unspecified: pause and escalate.
